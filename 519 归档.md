## 关键修复细节

### 1. cc-bridge Tier 2 拿不到 cwd

文件：`packages/desktop-shell/cc-bridge.cjs`

cc 2.1+ 每条 transcript jsonl 首行是元数据记录（`type: "permission-mode"` / `file-history-snapshot` 之类），不带 `cwd` 字段。原 `readFirstJsonl()` 只读首行，看到没 cwd 就 skip → 永远拿不到 D:\lll\pace 的 session。

修法：换成 `findCwdRecord(filePath, 32_768, 30)`（L134 / L190），扫前 30 行或 32KB 找第一条带 cwd 的记录。实际第 3 行（type=user 首条用户消息）才有 cwd。

证据：smoke 跑真实 `~/.claude/projects/D--lll-pace/<sid>.jsonl` → `cc_session_found=true`, `transcript_count=8`, `_meta.tier2=true`, `_meta.tier3=true`。

### 2. chatStream 没有 idle watchdog

文件：`packages/desktop-shell/llm-client.cjs`

之前只有绝对 `timeoutMs`（mentor-pipeline 设 120s）。MiniMax 偶发在长 context + tool result 回填后停止吐 token，但 TCP 连接没断 → 用户对着光标干瞪 120s 才拿到 timeout。

新增 `inactivityMs`（默认 30_000）。机制：

- 每个 socket byte + 每个 parsed delta 重置 `lastActivity`
- `idlePollMs = Math.min(5_000, Math.max(1_000, Math.floor(inactivityMs / 4)))` 周期检查（L261）
- 超时通过共享 `AbortController` abort
- 新 `error_code: 'stream_idle'`（L346），区分绝对 timeout 和 idle
- 新 i18n key `err.stream_idle.md`（i18n.cjs L237），zh+en 双语，用户看到「流式输出卡住了，重试同一个问题」
- `mentor-pipeline.cjs` L302 路由该 error_code 到 `errorMarkdown`

证据：

- mock SSE：1 chunk 后 park，`inactivityMs=1500` → 2016ms abort with `error_code='stream_idle'`
- 真实 MiniMax mentor turn 31s 含 `git_diff` 工具调用：clean 完成，watchdog 不误触，mentor 答案准确说出"你在修两个 bug"

### 3. TODO 一键 approve & run

新文件：`packages/desktop-shell/mentor-cmd-exec.cjs`（182 行）
关联：`main.cjs` +27 / `preload.cjs` +15 / `panel.html` +104 / `panel.js` +172 / `i18n.cjs` +12

`classifyCommand(cmd)` 返回 `{ class: 'safe' | 'caution' | 'deny' | 'unknown' | 'invalid' }`：

- `safe`：`git push|pull|fetch|status|log|diff|branch|show`、`npm install|ci|run X|test|ls|outdated`、pnpm/yarn、`ls/pwd`
- `caution`：`git commit|add|checkout|stash|merge|rebase|tag|restore`
- `deny`：`rm -rf` / `--force` / `--no-verify` / `sudo` / `curl|sh` 管道 / `drop table` / `git reset --hard` / `git push --force` / `git clean -*` / 重定向到 `/dev/*`
- `unknown`：未匹配
- DENY 先判定；multiline 或 >500 char 也归 deny

`execCommandStream(cmd, opts, onChunk)`：argv tokenize（无 shell 插值）、spawn、cwd 自动解析到 git root、60s timeout、stdout/stderr/exit 三类 chunk 经 IPC 推回 renderer。

panel.js 端 chip 4 类视觉差异：

| 分类 | 按钮文案（zh） | 行为 |
| --- | --- | --- |
| safe | ▶ 直接跑 | 点了即跑 |
| caution | ▶ 跑(需确认) | 弹 inline 确认行「确定要执行?」[✓ 执行] [✗ 取消] |
| deny | 🚫 拒绝执行 | disabled，title = matched dangerous pattern |
| unknown | ? 仅可复制 | disabled，title = not whitelisted |

执行时在 chip 下挂 `<pre.todo-output>` 流式渲染：stderr 琥珀色、exit 行末尾显示「exit 0 · 0.3s」（绿/红区分）。

CDP 全 4 类 demo：

- `git status --short` → safe，0.3s exit 0，真实 dirty file 列表流出来
- `git log --oneline -5` → safe（fix 后），0.3s exit 0，5 commit 列出
- `git commit -m "demo"` → caution，inline 确认行显示
- `rm -rf /` → deny，按钮 disabled，title 提示 dangerous pattern

权衡：只白名单的 shape 能一键跑；ad-hoc shell（curl 管道、find pipeline）仍只能 Copy。Worker session dispatch（直接派去 cc / codex worker）和 team-agent alignment 留在 v0.2。

### 4. IPC race（cmd-chunk 比 invoke result 晚到）

文件：`packages/desktop-shell/preload.cjs`、`mentor-cmd-exec.cjs`

`streamMentorAsk` / `execCommand` 原来都用 `ipcRenderer.invoke().finally(() => removeListener)`。当 exec 极快完成（`git status` <100ms），Electron 偶尔先把 invoke 的 return value 投递到 renderer、再投递入队列的 'exit' chunk。`.finally` 执行 → listener 被删 → 之后到的 exit chunk 无人接收 → chip 卡在「⏳ Running…」永不翻回。

修法：去掉 `.finally`。Handler 自己看 chunk type 是 exit/error 时 `setTimeout(removeListener, 50)`，50ms 留给 Electron scheduler 把 exit 之前队列里的 chunk 投递干净。两处都打补丁（execCommand + streamMentorAsk）。

附带顺手修了 `git log` regex：原 `/^git\s+log(\s+-\d+)?(\s+--oneline)?(\s+--stat)?$/` 顺序敏感，`git log --oneline -5`（最常见写法）匹配不上。新：`/^git\s+log(\s+(-\d+|--oneline|--stat|--all|--graph|--decorate))*$/` 接任意顺序任意组合。

### 5. DeepSeek 风格 thinking 框

文件：`packages/desktop-shell/panel.js`、`panel.html`

用户原话：「思考的那个框的内容太多了，最好是类似 DeepSeek 的一样：思考框最高多高，中间的内容新加的就往上刷新，等到正文到了再把思考框收起来放出下面的正文回复」

之前 thinking 框 `firstAnswerSeen` 时只加 `done` class（换 label + 关 brain animation），不加 `collapsed`，正文 stream 开始后思考框仍占大半个气泡。

(1) panel.js（L869-882）：`firstAnswerSeen` 时同时 add `collapsed` class（已有 CSS 规则 `.msg-thinking.collapsed .thinking-body { display: none }`）+ 挂 click-to-expand。L911-919 done handler 用 `dataset` guard 不重附 handler。

(2) panel.html：

- `.thinking-body` max-height 200 → 140（思考是配菜不是主菜）
- `mask-image: linear-gradient(to bottom, transparent 0, #000 28px, #000 100%)` 顶部 28px 渐隐，把视线引到底部最新行
- 4px accent-tinted 滚动条，整个框读起来像 ambient 而非 content region
- 暗色/亮色主题通用，不需 theme override

`scrollTop = scrollHeight` 自动滚到底的逻辑保留。

证据：CDP-driven dogfood 用 thinking-forcing prompt 跑完整 timeline——t=9.5s thinking 显示展开 / t=11.3-26.8s 思考字数增长 / t=29.2s 第一个 answer chunk 到达瞬间思考框折叠为 "140 字 · 29.3s" 一行 header。

### 6. MiniMax tool-call 协议泄漏

文件：`packages/desktop-shell/mentor-pipeline.cjs`

MiniMax-M2.7 偶发把 tool-call 协议外壳（应该走 OpenAI 兼容层结构化 `tool_calls` 字段）直接吐到 content 文本流，用户看到 mentor 回答里夹一段：

```
<minimax:tool_call><invoke name="git_status">...</invoke></minimax:tool_call>
```

`stripThinkBlocks()`（L414-435）扩展到 4 个 pattern：

```js
.replace(/<minimax:tool_call>[\s\S]*?<\/minimax:tool_call>/gi, '')  // 完整块
.replace(/<minimax:tool_call>[\s\S]*$/i, '')                         // 未关闭（OOM）
.replace(/<invoke\b[^>]*>[\s\S]*?<\/invoke>/gi, '')                  // 完整 invoke
.replace(/<invoke\b[\s\S]*$/i, '')                                   // 未关闭 invoke
```

未关闭形式从开标签洗到字符串结尾，保证最终答案 coherent。

两处 stream 路径里 inline 的 `answerAccum.replace(/<think>...<\/think>/, '')` 合并改用 `stripThinkBlocks()`（L598 / L850），共享 MiniMax 清洗逻辑。L743 retry 路径同步。

证据：5/5 smoke（complete block / unterminated tool_call / lone invoke / 混合 think+tool_call / 正常答案不动）。

Caveat：stream 期间 raw 标签仍可能短暂闪现为 text node，等 done chunk 替换为 stripped markdown 才彻底干净。要做到 stream 中零闪现需要给 SSE parser 加 sniff buffer，先 defer 到用户报怨明显闪现再做。

## 残留 / 未解决

1. **transcript user 视角缺失**：cc-bridge smoke 输出 transcript 4-8 条全是 assistant turn。cc 把 user input 的 `content` 存成数组/嵌套结构，`readTranscriptExcerpt` 没解出来——mentor 拿到的对话视角片面。下一步可能要扩 transcript parser。
2. **stream 期间 raw tag 短暂闪现**：MiniMax tool-call leak fix 只在 done 时清洗，stream 中实时 text node 可能闪 raw 标签 1-2 帧。要彻底无闪现需 SSE parser 加 sniff buffer at the answer boundary。
3. **mentor pipeline 偶发挂死的 root cause**：MiniMax 长 context + tool result 回填后偶发停吐 token 也不发 `[DONE]`。inactivity watchdog 现在能兜底（30s 后用户看到友好错误而非死等 120s），但 server 端原因未查清。

## v0.2 拟入项

1. **Worker dispatch**：把 caution/safe TODO 直接派到 cc 或 codex worker session 跑（用户原话：「可以直接让 cc 或者 codex 的 worker session 去执行」），需要先有 worker registry。
2. **Team-agent alignment**：根据同事 commit 直接和同事的 agent 对齐消息/进度（用户多次提到的核心 PMP-style 场景之一）。
3. 上两项都从今天的 approve & run feature 长出来——今天落地的是「自己进程里 spawn whitelist-safe」，v0.2 把目标进程换成 worker session / 远端 agent。
