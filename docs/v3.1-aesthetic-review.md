## 3 大痛点（按严重度排序）

### 痛点 1 · 动效全部停在终态（"幻灯片化"）—— 维度 5 4/10 的根因

**证据帧**：
- `sweep-56s.png`：mentor 答案 "建议下一步：" 列表项 0/4 显示——step stagger delay 排到 5.0s，进入下个 segment 时还没开始
- `sweep-64s.png`：mentor 答案 4 行 list 完整显示但**没有任何动画进行**，观众看到的是终态截图
- `sweep-72s.png`：Team tab 3 张卡片全部到位 + Tom hover 动效已结束（hoverPop 4.0s 是 segment 末尾）
- `sweep-88s.png` + `sweep-96s.png`：两张帧**完全一样**——Tom 答案 3 段已全部 fade-in 完成，但相隔 8s 画面零变化
- `frame-108s.png`：红色 "它不是 task tracker" 静态，红线已划完，没看到划线过程

**根因诊断**：v3.1 storyboard 几乎所有 `@keyframes` 都用 `forwards` filling mode + 单段 0% → 100% transition + 一次性 delay。结果是每个 segment 在前 1-2 秒做完动效就**停在终态长达 5-10 秒**。Beat 3（41s）实际只有前 7 秒有运动，后 34 秒是静止画面。

**反 pitchkit-video-design §动效编排** "有编排感：一次 staged reveal 胜过满屏元素各动各的；转场一种为主、克制；不 PPT（全静止）"——v3.1 是后者：PPT 化。

**升级方向**（v4）：每个 segment 内**至少 5 个动效里程碑**均匀分布到时长全程，不允许超过 3 秒静止画面。如 Beat 3 41s 内必须有 ≥ 12 个独立 keyframe 里程碑（工具行砸入 × 3 + thinking 计数器累积 × 4 + morph 中间帧 × 3 + bloom 进行中 × 2）。

### 痛点 2 · 转场低级（fade 之外全是黑屏）—— 维度 5 + 维度 3 共因

**证据帧**：
- `sweep-40s.png`：**画面 99% 黑屏**，只有右下角凭据角注 "查 5 个 commit · 47 行改动 · 8 轮对话" 可见。这是 transcript morph segment 进入瞬间（fade_in 12 帧 + tool1 stagger 0.6s 重叠的结果），观众看到的是 "ttt 黑屏 + 一行小字漂浮"
- `sweep-48s.png`：完全黑屏（只有右下角 credits）——是 segment 5 进入与 segment 6 fadeBlur 收尾的衔接黑洞
- `sweep-80s.png`：双轨切换瞬间又全黑——splitOut 1.0s 完成后到 fullIn 1.35s 之间 350ms 真空帧

**根因诊断**：v3.1 全片转场配额 = fade 主 + 1 wipe（76s）+ 1 spatial bloom（52s）。bloom 和 wipe 这两个**强转场**理应是高光时刻，但抽帧看 spatial bloom 完全看不见（52s sweep 没抽到 600ms 的 bloom 窗口），wipe-left 是 350ms cubic-bezier 但前后留了 350ms 黑屏间隙。其余 11 处 segment-to-segment 转场都是 fade 0.4s——剪辑师 OK，观众体感是 "PPT 翻页"。

**反 pitchkit-video-design §转场配额** "fade 主 + 至多 1 处非 fade"是上限不是平均值——意思是**那 1 处非 fade 必须做得让人记住**。v3.1 的非 fade 反而被压缩到看不见。

**升级方向**（v4）：
- Beat 3 内部 transcript morph（39-50s）用 **crossfade-via-blur** 真正长出来，每 1 秒一个可见的中间帧（左侧 blur 20% → 40% → 60% → 80% → 完全消失，同步右侧 typewriter 0% → 25% → 50% → 75% → 100%）
- Beat 3 → Beat 4 的 1 帧黑屏 + spatial bloom 改为 **2 帧黑屏 + bloom 800ms**（v3.1 是 1 帧 + 600ms，剪辑师能看到但 30fps 渲染抽帧抓不到任何中间态）
- 加 **box-shadow blast** 作为第 4 种转场（紫光晕从一点扩散到全屏 800ms）放在 Beat 1 → 2 的衔接

### 痛点 3 · 没展现产品核心（卖点没被可视化）—— 维度 7 5/10 的根因

**证据帧 + 缺失对照**：

| 产品核心 | 剧本承诺 | 实际抽帧 |
|---|---|---|
| Pace 读 cc transcript | "读最近 8 轮 cc 对话" | `sweep-32s.png`：只显示一行 mono 字幕"cc_recent_transcript → 读最近 8 轮 cc 对话"，**真实 transcript jsonl 在 39-50s morph segment 才出来一闪而过**（sweep-40s 完全黑屏） |
| 读 git_log 5 个 commit | "翻最近 5 个 commit" | 整片**没有任何一帧显示出 5 个 commit hash**。剧本只承诺"5 个 commit"是数字字幕，没有真实 hash 列表 |
| thinking 累积到 582 字 | 思考头部 "582 字 · 9.2s" | `sweep-32s.png`：直接显示终态 "582 字 · 9.2s"，**计数器没有从 0 累积到 582 的过程**——是死字幕不是计算过程 |
| mentor bubble 紫色脉冲 | mentor 左侧 2px 紫边 | 静态 `border-left: 2px solid #6B4FE0`，**不脉冲不呼吸**，与普通卡片左边框无区别 |
| 4 行肯定句逐行 typewriter | `stagger 350ms × 4 行` | `sweep-112s.png`：4 行**同时全部显示**，stagger 在抽帧采样间隔（8s）外，**观众看到的是 PPT 列表，不是逐行流字** |

**反 pitchkit-video-design §what 不是 how** "demo 视频核心是**让观众看到产品在动**，不是把功能名字打在字幕上"——v3.1 把所有产品能力写成了**字幕断言**（"读 cc"、"翻 git"、"think 582 字"），但**没有把能力可视化成画面**。结果是看了 117s 不知道 Pace 实际在屏幕上做了什么。

**升级方向**（v4）：
- Beat 3 工具调用阶段：**真显示 5 个 commit hash 列表**（用 Pace 仓库真实 `4ae4781 / b4f0758 / 04ac79d / d9fdd41 / 98d100e`），每行 200ms 滚出
- thinking 计数器：从 0 → 582 用 setInterval 60ms 步进可见的数字滚动 9.2s，不是死字幕
- transcript morph：左侧真 jsonl 多行（最少 6 行 cc session 真实结构），每行从下往上 fade out + blur，**同步右侧 typewriter** 一字一字出干净答案
- mentor bubble：左边框 2px 紫边做 `box-shadow 0 0 0 0 → 8px purple-30% → 0` 1.2s 无限循环 breath
- 4 行肯定句：每行字符级 typewriter 50ms/字，整体 stagger 800ms 一行——确保单帧抽样必能命中 2-3 行进行中

## 结论

**v3.1 不可交付**——总分 6.0/10 离门槛 8.0/10 有 2 分缺口，动效维度 4/10 是结构性问题不是参数微调。

**根本原因**：v3.1 storyboard 把"动效"当成"动效声明"——`@keyframes` 写了 50+ 个，但每个都是单次 0%→100% + forwards，导致**绝大多数画面停在终态**。观众看到的是"剧本里承诺的画面"的截图序列，不是"产品在动"。

**改 v4 的最关键 2 件事**：
1. 每个 segment 内强制 ≥ 5 个 keyframe 里程碑均匀分布，最长静止帧 ≤ 3s
2. 把产品核心能力从字幕断言转成可视化镜头：5 commit hash 真出现、thinking 计数器真滚动、transcript morph 真有中间帧、mentor 紫边真脉冲

v4 剧本（`demo-script-v4.md`）按这 2 条 + 维度 7 七个维度逐一回推具体动效 spec。
